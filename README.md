## 为什么很多团队做了灰度，还是不敢发版

很多团队并不是没有“灰度能力”，而是没有形成一套完整的发布体系。

常见情况是：

- 有流量切分，但没有全链路版本一致
- 有测试环境，但环境成本越来越高
- 有分支策略，但和发布流程脱节
- 有上线流程，但关键动作仍然依赖人盯人
- 有问题能回滚，但回滚仍然慢、重、风险大

最后的结果往往是：

- 发版还是紧张
- 出问题影响还是大
- 协作成本还是高
- 规范落不了地

所以真正需要的，不是一个零散功能，而是一套**平台级灰度发布全案解决方案**。

---

## 这套方案解决什么问题

这套方案面向的是企业在发布环节最核心的几类问题：

- 如何把全量发布改造成分批放量，降低单次上线风险
- 如何把回滚从重新部署，升级成规则级秒回退
- 如何让 `HTTP`、`MQ`、`RPC` 在跨服务场景下保持同一版本链路
- 如何减少测试环境重复建设，优化环境和维护成本
- 如何把项目管理规范、分支管理规范、平台能力和流程卡口统一起来

它解决的不是单点技术问题，而是发布过程中的**稳定性、效率、成本和治理**问题。

## 方案包含哪些内容

### HTTP 灰度发布

- 支持按比例、白名单、角色、用户属性等方式分批放量
- 支持渐进式发布，从小流量验证到逐步全量
- 支持关闭灰度规则后快速回到基线版本

### MQ 灰度发布

- 支持异步链路透传灰度标识
- 保证消息处理过程中的版本一致性
- 避免入口和异步消费版本错位

### RPC 灰度发布

- 支持服务调用链路中的灰度上下文透传
- 保持多服务联动时的版本一致
- 降低跨服务发布时的联动风险

### 项目管理规范

- 灰度准入评估
- 发布前检查项
- 演练机制
- 复盘机制
- 审计通知闭环

### 分支管理规范

- 基线版本管理
- 灰度分支生命周期管理
- 灰度成功后的主干合并机制
- 测试环境基线同步机制
